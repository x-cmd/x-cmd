# Score

"Don't guess. Score."

加权维度评分框架，用于 AI agent 辅助决策。

## 如何使用

**先问够，再定标准。** 在定义维度之前问够问题。维度搞错了，分数再精确也没用。

**抛砖引玉，迭代优于空想。** 需求不清楚时，先做一版出来，打分排名，给用户看。用户对具体排名有天然反应——"这个权重不对"、"你漏了 XX"。用 `x score refine` 调整，重新打分。一轮迭代比十轮空想更高效。

**Agent 是分析师，不是计算器。** `compute` 之后读 TSV——解释为什么 A 赢了 B、有什么意外、建议调整什么。排名是对话的开始，不是结束。

## 核心流程

```
① 了解需求      → 问清楚用户要什么，场景是什么
② 制定标准      → 写 score.yml（维度 + 因子 + 评分描述）
③ 打分          → 写 score.tsv（目标列表 + AI 打分 0–10 + 证据）
④ 核算校验      → x score compute <score.yml> <score.tsv>
⑤ 汇报与迭代    → agent 读排序结果，解释结论，根据反馈 refine
```

输出 TSV 固定列为 `target, total, rank, reason`，然后是带权重百分比标注的维度列（如 `成本 (35%)`），最后是 `evidence`。

总分 0–100。按 total 降序排列，rank = 1 为最高分。

## 例子 — 协助用户进行宣传口号方案决策

1. 先与用户商量，得出评价维度与标准，生成 `宣传口号.score.yml`
2. 决策
    - 如果有 x-cmd , 可以 x score assess -f "<宣传口号>.score.yml" [...targets-to-assess] , 会自动生成文件 <宣传口号>.<datetime like 260605-194400>.score.tsv
    - 或者，可以执行以下等同步骤:
        - 2a: 将待决策的口号列出放在第一列，让 AI 对每个口号在各维度上打分（0–10，附 evidence），生成 `宣传口号.260605-194900.score.tsv`。总分和 rank 留白，等待步骤 2c 计算
        - 2b: 通过 `compute.py` 或 `x score compute` 计算 total + rank，补全 `宣传口号.260605-194900.score.tsv`
3. 读取结果，与用户讨论。可调整步骤或者增加候选后重复步骤 2
    - 可以用 x score iter -f "<宣传口号>.score.yml" `宣传口号.260605-194900.score.tsv` "<调整意见，或者增加候选>"

以上文件名仅作示意，实际可自行命名。

更多例子，参见 [EXAMPLE.md](EXAMPLE.md)。

## 两文件模型

| 文件 | 角色 |
|------|------|
| `*.score.yml` | 可以复用的评分标准：维度、因子、描述 |
| `**.score.tsv` | 目标 + 每个维度的分数与证据；第一列为目标名 |

如果装了 x-cmd：`x score compute <yml> <scores>` 等价于 `python compute.py`。

## 应用场景

1. **给 agent 装上决策工作流。** score.yml 是 agent 的决策锚点——不再随上下文左右摇摆。agent 按标准走：提问 → 打分 → compute → 汇报。

2. **决策有规则可依。** 维度 × 因子 × 证据 → 加权总分 → 排名。每个分数锚定在事实上，推理过程透明可复盘。

3. **随需求灵活微调。** 换维度、调因子、加硬过滤——同一个模板适配不同用户，改的是重视什么、有多重视。

4. **随需求清晰而进化。** 初版抛砖引玉 → 用户反馈 → refine → 重新打分 → 越来越准。

5. **任何 skill 都能拥有标准文件 + 存档 + 迭代。** score.yml = 决策标准。score.tsv = 决策记录。两份文件存档后可复盘、可分享、可改进。

## 公式

```
total_i = Σ(f_j × score_ij) ÷ Σ(f_j) × 10
```

- **f_j** — 维度 j 的因子（整数 1–10）。基线 = 2，越大表示越重要。
- **score_ij** — 目标 i 在维度 j 上的分数（0–10）。
- **total_i** — 目标 i 的加权总分（0–100）。

按总分降序排列，rank = 1 为最高分。

因子是放大倍数，会自动归一化。基线 = 2（正常权重）。不重要的维度给 1。按倍数上调：4 = 2×、6 = 3×、8 = 4×、10 = 5× 重要度。因子不需要加起来等于任何值——公式会自动归一化。每个维度在输出 TSV 的列头会标注权重占比：`factor ÷ Σ(factor) × 100`。

## 输出列

| 列 | 来源 | 说明 |
|----|------|------|
| `target` | 输入 | 候选名称 |
| `total` | 计算 | 加权总分（0–100） |
| `rank` | 计算 | 排名（1 = 最高）。`X` = 被 block 淘汰 |
| `block` | 输入 | 可选的硬过滤列。非空 = 该候选被淘汰，排到最后，rank 显示 X。 |
| `reason` | 输入/输出 | 可选：AI 标注评分的 nuance |
| `<维度> (X%)` | 计算 | 各维度分数，列头显示权重占比 |
| `证据` | 输入 | 支撑每个分数的事实依据 |

## 分数刻度

| 分数 | 标签 | 含义 |
|------|------|------|
| 9–10 | 优秀 | 完全符合维度标准 |
| 7–8 | 良好 | 符合标准，有微小差距 |
| 5–6 | 及格 | 方向正确，有明显不足 |
| 3–4 | 差 | 差距显著 |
| 1–2 | 失败 | 几乎没有达到 |
| 0 | 不适用 | 不适用 |

## 评分规则

1. **证据锚定** — 每个分数必须有证据。证据是事实：字母数、用户百分比、基准测试结果。"感觉不错"、"直观"、"看起来对"不是证据。
2. **用全范围** — 5 是中间点，不是默认分。0 分和 10 分少见但可以给。
3. **因子是相对放大器** — 基线=2。因子 4 的影响力是因子 2 的两倍。因子不需要求和。
4. **目标唯一且不能为空** — `**.score.tsv` 第一列每行都必须有目标名，不可重复，不可为空。
5. **Agent 读结果、汇报、决策** — `compute` 后 agent 读取排序 TSV，向用户汇报结论。如果结果有问题，agent 可以重新打分、调整因子、或增减候选。`reason` 列用于传递 agent 间的备注信息。

## 文件清单

| 文件 | 用途 |
|------|------|
| `SKILL.cn.md` | 中文版。自包含的评分引擎。AI 读取后即可运行完整工作流。 |
| `SKILL.md` | 英文版。 |
| `compute.py` | 独立 Python 校验器。`python compute.py <yml> <tsv>` 校验并计算 total + rank。 |
| `template/*.score.yml` | 模板文件。`x score ls` 列出，`x score init -t <name>` 复制。模板内含 comment 引导 agent 根据用户需求调整标准——格式见 `score-template-writer.md`。 |
| `rule-vs-score.md` | Rule 与 Score 的区别——什么场景用哪个，各自解决什么问题。 |
| `score-template-writer.md` | 模板编写指南。怎么写 comment、怎么选维度、怎么定 factor、怎么写出好的 desc。 |
